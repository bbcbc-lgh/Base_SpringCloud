# 答辩讲稿与高频问题

## 3-5 分钟讲稿

各位老师好，我的项目题目是《基于 Spring Cloud Alibaba 的微服务电商订单管理系统》。

这个项目围绕电商下单场景设计，主要实现了用户注册登录、Token 鉴权、库存查询、订单创建、库存扣减、订单消息通知和消息记录查询。系统采用微服务架构，拆分为 gateway、auth-service、user-service、order-service、stock-service、message-service、monitor-service 和 common 公共模块。

系统入口是 Spring Cloud Gateway，客户端所有请求先进入网关，再根据路由转发到对应服务。认证服务负责注册登录和 Token 签发，用户服务负责用户数据管理，订单服务负责创建订单，库存服务负责库存查询和扣减，消息服务负责消费订单消息并保存消息记录。

项目使用 Nacos 做服务注册发现，各个微服务启动后会注册到 Nacos，服务之间通过 OpenFeign 进行远程调用。比如订单服务创建订单时，会通过 Feign 调用库存服务扣减库存。为了保证订单和库存的一致性，订单创建方法上使用了 Seata 的 `@GlobalTransactional` 开启全局事务，如果库存扣减失败，订单创建也会回滚。

订单创建成功后，订单服务会向 RocketMQ 发送订单创建事件，消息服务作为消费者监听该消息并写入消息记录表。这样可以把订单主流程和消息记录解耦，提高系统扩展性。系统还接入了 Sentinel，用于限流和服务保护；Redis 用于缓存登录 Token；SkyWalking 用于链路追踪展示。

我使用 Postman 对注册、登录、查询用户、查询库存、创建订单、查询订单和查询消息记录做了接口测试，也做了未携带 Token、库存不足等异常测试。使用 JMeter 对登录、查询库存和创建订单做了压力测试，在 10 个线程、循环 5 次的条件下，总请求 150 次，错误率为 0，说明核心业务链路在本地环境下运行稳定。

项目的亮点主要有三个：第一，微服务拆分清晰，认证、用户、订单、库存、消息职责独立；第二，Spring Cloud Alibaba 组件覆盖较完整，包括 Nacos、Gateway、Feign、Seata、RocketMQ、Sentinel；第三，完成了从功能验证到接口测试、压力测试的完整证据链。

## 高频问题答案

### 1. 为什么使用 Nacos？

Nacos 用于服务注册发现和配置管理。微服务启动后注册到 Nacos，调用方可以通过服务名访问目标服务，不需要写死具体 IP 和端口，便于服务扩容和维护。

### 2. Gateway 有什么作用？

Gateway 是系统统一入口，负责请求路由、跨域处理、鉴权信息透传等。客户端只需要访问 Gateway，不需要直接访问多个后端服务。

### 3. Feign 怎么调用服务？

Feign 通过声明式接口调用远程服务。项目中订单服务通过 `StockFeignClient` 调用库存服务的扣减接口，认证服务通过 `UserFeignClient` 调用用户服务。

### 4. Seata 解决什么问题？

Seata 用来解决分布式事务一致性问题。订单创建涉及订单表写入和库存扣减，这两个操作分布在不同服务中。使用 Seata 后，如果库存扣减失败，订单写入也会回滚。

### 5. RocketMQ 为什么适合订单消息？

订单创建后的一些后续动作，比如消息记录、通知、日志等，不一定要阻塞主流程。RocketMQ 可以异步处理这些任务，提高系统解耦性和扩展性。

### 6. Sentinel 怎么保护服务？

Sentinel 可以配置限流规则，当访问量过大时对请求进行限制，避免服务被瞬间流量压垮。项目中各服务都配置了 Sentinel Dashboard 地址，并在公共模块中提供限流规则配置。

### 7. Redis 在项目中有什么作用？

Redis 主要用于缓存登录 Token。用户登录后，Token 会保存到 Redis，后续鉴权可以快速校验，也便于后续扩展 Token 失效和退出登录能力。

### 8. 项目还有哪些不足？

目前项目主要面向课程演示，后续可以继续完善自动化测试、Docker Compose 一键部署、OpenAPI 接口文档、日志采集告警、配置中心隔离和生产级安全配置。
